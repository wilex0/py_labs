# Лабораторная работа №4: Интерфейсы и абстрактные классы (ABC)

## Цель работы

Изучение и практическое применение абстрактных базовых классов (ABC) для создания интерфейсов, определяющих контракты поведения. Освоение полиморфизма через интерфейсы и интеграция с коллекцией объектов из ЛР-2.


## Описание интерфейсов

### 1. Updatable (Обновляемый)

Объекты с возможностью автоматического обновления состояния.

**Абстрактный метод:**
- `update() -> None` — обновляет состояние объекта

**Реализация в классах:**
- `FoodProduct` — обновляет скидку в зависимости от оставшихся дней до истечения срока годности. Если продукт просрочен — обновление не происходит
- `DigitalProduct` — обновляет скидку в зависимости от текущего рейтинга. Если нет отзывов или подходящих скидок — обновление не происходит
- `Product` — не реализует (базовый продукт не имеет критериев для обновления)

### 2. Printable (Печатный)

Объекты с возможностью строкового представления и вывода истории изменений.

**Абстрактные методы:**
- `to_string() -> str` — возвращает строковое представление объекта
- `print_history() -> None` — выводит историю изменений объекта

**Реализация в классах:**
- `Product` (базовый) — `to_string()` делегирует вызов `__str__()`, `print_history()` выводит историю изменения цен
- `FoodProduct` — добавляет в `to_string()` информацию о сроке годности, в `print_history()` — историю скидок по сроку годности
- `DigitalProduct` — добавляет в `to_string()` информацию о рейтинге, в `print_history()` — историю отзывов и скидок по рейтингу

### 3. Comparable (Сравнимый)

Объекты с возможностью сравнения между собой.

**Абстрактный метод:**
- `compare_to(other) -> int` — сравнивает текущий объект с другим. Возвращает -1 (хуже), 0 (равны), 1 (лучше)

**Реализация в классах:**
- `FoodProduct` — сравнение по свежести:
  1. Просроченный продукт всегда хуже не просроченного
  2. Сравнение по дням до истечения срока (больше дней = лучше)
  3. При равенстве дней — сравнение по цене (дешевле = лучше)
  4. Выбрасывает `TypeError` при сравнении с объектом не-FoodProduct

- `DigitalProduct` — сравнение по рейтингу:
  1. Сравнение по рейтингу (выше рейтинг = лучше)
  2. При равенстве рейтинга — сравнение по количеству отзывов (больше = лучше)
  3. При равенстве количества отзывов — сравнение по цене (дешевле = лучше)
  4. Выбрасывает `TypeError` при сравнении с объектом не-DigitalProduct

- `Product` — не реализует (базовый продукт не имеет критериев для сравнения)


## Интеграция с коллекцией ProductCollection

Коллекция из ЛР-2 расширена методами для работы с интерфейсами.

### Методы фильтрации по интерфейсам

- `get_printable() -> ProductCollection` — возвращает новую коллекцию из объектов, реализующих `Printable`
- `get_updatable() -> ProductCollection` — возвращает новую коллекцию из объектов, реализующих `Updatable`
- `get_comparable() -> ProductCollection` — возвращает новую коллекцию из объектов, реализующих `Comparable`

### Методы массовых операций

- `update_all() -> None` — обновляет все объекты в коллекции, реализующие `Updatable` (полиморфный вызов `update()`)
- `print_all() -> None` — выводит все объекты коллекции через `Printable.to_string()`
- `print_all_history() -> None` — выводит историю всех объектов через `Printable.print_history()`

### Методы сортировки и поиска

- `sort_by_comparable(reverse=False) -> None` — сортирует коллекцию, группируя объекты по типам и сортируя каждую группу через `compare_to()`. Объекты, не реализующие `Comparable`, перемещаются в конец списка
- `find_best() -> list[Product] | None` — находит лучшие продукты среди сравнимых (по одному для каждого типа: FoodProduct и DigitalProduct)
- `find_worst() -> list[Product] | None` — находит худшие продукты среди сравнимых (по одному для каждого типа)

## Сценарии демонстрации (demo.py)

В демонстрации создается коллекция из 9 продуктов:
- 2 базовых продукта (`Product`) — реализуют только `Printable`
- 4 продукта питания (`FoodProduct`) — реализуют все три интерфейса
- 3 цифровых продукта (`DigitalProduct`) — реализуют все три интерфейса

### Сценарий 1: Фильтрация коллекции по интерфейсам

Демонстрирует методы `get_printable()`, `get_updatable()`, `get_comparable()`:
- `get_printable()` возвращает все 9 продуктов
- `get_updatable()` возвращает 7 продуктов (FoodProduct и DigitalProduct)
- `get_comparable()` возвращает 7 продуктов (FoodProduct и DigitalProduct)
- Проверка реализации интерфейсов через `isinstance()` для трех разных типов продуктов
- Сводная таблица с количеством объектов каждого типа, реализующих каждый интерфейс

### Сценарий 2: Массовое обновление объектов

Демонстрирует применение скидок через интерфейс `Updatable`:
- Отображение состояния продуктов до обновления (скидки 0%)
- Выполнение `update_all()` для всех обновляемых объектов
- Отображение состояния после обновления (появляются скидки у продуктов с подходящими условиями)
- Наглядное объяснение разного поведения `update()`:
  - `FoodProduct` — применяет скидку 20% на молоко (осталось 2 дня из 3 допустимых)
  - `DigitalProduct` — применяет скидку 15% на Python курс (рейтинг 4.5 >= порога 3.5)

### Сценарий 3: Сравнение объектов

Демонстрирует метод `compare_to()` интерфейса `Comparable`:
- Сравнение продуктов питания по свежести (milk_fresh vs bread, cheese vs milk_fresh, свежее vs просроченное)
- Сравнение цифровых продуктов по рейтингу (python_course vs photo_editor, photo_editor vs ebook_basic)
- Попытка сравнения разных типов (FoodProduct vs DigitalProduct) — ожидаемая ошибка `TypeError`

### Сценарий 4: Сортировка коллекции

Демонстрирует сортировку через интерфейс `Comparable`:
- Сортировка всей коллекции через `sort_by_comparable()` — продукты питания сортируются по свежести, цифровые по рейтингу, базовые остаются в конце
- Поиск лучших и худших продуктов через `find_best()` и `find_worst()` (возвращают по одному лучшему/худшему для каждого типа)
- Отдельная сортировка только продуктов питания (от свежего к просроченному)
- Отдельная сортировка только цифровых продуктов (от высокого рейтинга к низкому)

### Сценарий 5: Полиморфизм через интерфейсы

Демонстрирует работу с объектами через интерфейсы без привязки к конкретным типам:
- Сравнение плохого подхода (проверка `isinstance`) и хорошего (вызов `to_string()` через интерфейс `Printable`)
- Попытка обновления объектов через `Updatable` — базовый `Product` пропускается, так как не реализует интерфейс
- Вывод истории через `Printable.print_history()` — разное содержимое для разных типов
- Цепочка полиморфных операций: фильтрация → обновление → вывод
